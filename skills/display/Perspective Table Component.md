# Perspective Table Component

## Description

This guide explains the configuration of the Perspective Table component, a versatile tool for displaying interactive data. It details how to structure source data, define column properties, apply conditional styling, and utilize advanced rendering modes like embedded views or expandable subviews. The instructions also cover how to enable and respond to user interactions such as sorting, selection, and cell editing using component events and functions.

## Documentation

# Instructions
### Perspective Table Component

The Perspective Table component is a versatile tool used to display data in a tabular format. It's highly customizable, allowing for control over data presentation, styling, user interactions, and more. The primary source of data is typically a dataset or an array of objects.

---

### Core Concepts

#### Data Structure (`props.data`)

The `data` property is the heart of the Table. It can accept data in several formats:
1.  **Array of Objects (Recommended):** This is the most flexible format. Each object in the array represents a row, and each key-value pair in the object represents a cell in that row.
    *Example:* `[{ "city": "Folsom", "population": 77271 }, { "city": "Madrid", "population": 3233527 }]`
2.  **Dataset:** A standard Ignition dataset, often from a Tag or a database query.
3.  **Array of Arrays:** Each inner array represents a row. This format is less common and requires columns to be configured by index.

To control styling or editability on a **per-cell basis**, the value for that cell in the data source must be an object with a specific structure:
```json
{
  "value": "The actual cell value",
  "editable": true,
  "style": {
    "backgroundColor": "#F7901D",
    "classes": "some-class"
  }
}
```
This structure is most easily achieved by using an "Array of Objects" as the main data format and applying a Script Transform if the original data is not in this shape.

#### Column Configurations (`props.columns`)

The `columns` property is an array of objects where each object configures a single column in the table.

**CRITICAL NOTE:** If you configure **any** column by adding an object to the `props.columns` array, the table will **only** display the columns you've explicitly defined in this array. You must add an entry for every column you wish to see.

Each column configuration object has several key properties:

*   `field`: This is the most important property. It maps the column configuration to a key in your `props.data` objects (e.g., "city", "population").
*   `visible`: A boolean that toggles the column's visibility. The data is still present but hidden from the user.
*   `render`: Dictates how the cell's data is displayed. Can be `auto`, `number`, `date`, `boolean`, `string`, or `view`.
*   `editable`: A boolean to make all cells in the column editable. This can be overridden by the `editable` property on an individual cell's data object.
*   `sortable`: A boolean to allow users to sort the table by this column.
*   `width`: Sets the column's width. Can be a proportion of available space or a fixed pixel value if `strictWidth` is true.
*   `header` & `footer`: Objects to configure the column's header and footer cell, including `title` and `style`.

---

### Key Features and Properties

#### Rendering Modes (`columns.render`)

The `render` property in a column's configuration unlocks powerful display options:

*   **`number`**: Renders the value as a number.
    *   Use `numberFormat` to specify the format (e.g., "0,0.##", "$0,0.00").
    *   Set the `number` property to `progress` to display a progress bar instead of text. Configure the bar using the `progressBar` object (`min`, `max`, `bar.color`, etc.).
*   **`date`**: Renders the value as a date/time.
    *   Use `dateFormat` to specify the format (e.g., "MM/DD/YYYY", "h:mm:ss a").
*   **`boolean`**: Renders a boolean value.
    *   Use the `boolean` property to choose the display style: `checkbox` (default), `toggle`, or `value`.
    *   Configure the `toggleSwitch.color` object for custom selected/unselected colors.
*   **`view`**: Renders a separate Perspective View inside each cell of the column. This is extremely powerful for custom visualizations.
    *   `viewPath`: The path to the Perspective View to embed.
    *   `viewParams`: An object of parameters to pass to the embedded view.
    *   **Implicit View Parameters:** The Table automatically passes the following parameters to each cell view, which can be used to configure components within that view:
        *   `value`: The value of the specific cell.
        *   `row`: The true index of the row from the source `props.data`.
        *   `rowIndex`: The visible index of the row as displayed (accounts for paging and filtering).
        *   `column`: The field name of the column.
        *   `columnIndex`: The index of the column.
        *   `rowData`: A JSON object containing all the data for the entire row.

#### Row Subviews (`props.rows.subview`)

You can configure rows to be expandable, revealing a subview with additional details.

1.  **Enable:** Set `props.rows.subview.enabled` to `true`. An expand icon will appear on each row.
2.  **Select View:** Set `props.rows.subview.viewPath` to the path of the View you want to show when a row is expanded.
3.  **Pass Parameters:** Use `props.rows.subview.viewParams` to pass custom parameters.
4.  **Implicit Subview Parameters:** Like cell views, subviews automatically receive parameters from the row they belong to. **The subview must have a View Parameter named `value` of type Object to receive the row's data.**
    *   `value`: A JSON object representing all the data for the expanded row (e.g., `params.value.city`, `params.value.population`).
    *   `row`: The true index of the row from `props.data`.
    *   `rowIndex`: The visible index of the row.

#### Conditional Styling

To change the style of a row or cell based on its data (e.g., color a row red if a value is too high), you must modify the `props.data` structure. This is almost always done using a **Script Transform** on a property binding to `props.data`.

**Method:**
1.  Bind the Table's `props.data` property to your source of data (e.g., a custom property or a Tag).
2.  Add a Script Transform to the binding.
3.  The script should iterate through the incoming data (`value`) and build a new list of objects.
4.  For each row, determine the style based on your condition.
5.  For each cell in that row, create a dictionary: `{"value": original_value, "style": style_object}`.
6.  Return the new, fully-formatted list of objects.

**Example Script Transform to Style an Entire Row:**
```python
def transform(self, value, quality, timestamp):
    newData = []
    # value is assumed to be a list of dictionaries
    for row in value:
        # Define a style based on a condition
        if row.get("population") > 1000000:
            style = {"backgroundColor": "#FFCCCC"} # Light Red
        else:
            style = {} # No special style

        # Rebuild the row with value/style objects for each cell
        newRow = {}
        for key, cellValue in row.items():
            newRow[key] = {"value": cellValue, "style": style}
        newData.append(newRow)
        
    return newData
```

#### User Interaction

*   **Sorting:** If `columns.sortable` is true, double-click a header to sort. Hold **Shift** and double-click multiple headers for multi-column sorting. The initial sort order can be set with the `props.sortOrder` array.
*   **Selection:** Controlled by the `props.selection` object.
    *   `mode`: Can be `single`, `single interval`, or `multiple interval`.
    *   `enableRowSelection` / `enableColumnSelection`: Control whether users select rows, columns, or individual cells (if both are true).
    *   The current selection is available in `selection.data`, `selection.selectedRow`, and `selection.selectedColumn`.
*   **Editing:** If a column is `editable`, users can edit cells by an action defined in `props.cells.allowEditOn` (e.g., `double-click`). The edit must be committed with the **Enter/Return** key.
*   **Paging:** Controlled by the `props.pager` object. Enable the top or bottom pager, set page size options, and see the current page.

---

### Component Events

Component events allow you to run scripts or actions in response to user interaction.

*   `onEditCellCommit`: Fires after a user successfully commits an edit. The `event` object contains:
    *   `event.column`: The field name of the edited column.
    *   `event.row`: The source data index of the edited row.
    *   `event.value`: The new, committed value.
    *   **Use Case:** This is the ideal event for writing the new value back to a database or Tag.

*   `onSelectionChange`: Fires when the user's selection changes. The `event` object contains:
    *   `event.selectedColumn`: The currently selected column's field name.
    *   `event.selectedRow`: The currently selected row's index.
    *   `event.data`: An array of objects representing the selected data. The structure of the objects depends on the selection mode.

*   `onRowClick` / `onRowDoubleClick`: Fires when a row is clicked or double-clicked. The `event` object contains:
    *   `event.value`: An object with all the data for the clicked row.
    *   `event.row`: The source data index.
    *   `event.rowIndex`: The visible row index.

*   `onSubviewExpand` / `onSubviewCollapse`: Fires when a subview is opened or closed. The `event` object contains the same information as the row click events.

---

### Component Functions

You can call these functions on the component instance in scripts (e.g., `self.getSibling("Table").collapseSubviews()`).

*   `.collapseSubviews([rowIndices])`: Collapses expanded subviews. If the optional `rowIndices` array is provided, only those specific rows are collapsed. Otherwise, all subviews are collapsed.
*   `.expandSubviews([rowIndices])`: Expands subviews for the specified rows. The rows must be visible on the current page.

---

### Helpful Tips & Best Practices

*   **Row vs. RowIndex:** Be very careful to distinguish between `row` and `rowIndex` in events and view parameters. `row` is the absolute index in your source `props.data`. `rowIndex` is the visual index on the screen, which is affected by sorting, filtering, and paging. Use `row` when you need to modify the source data.
*   **Data Format:** While the table can handle datasets, using an array of objects (`List<Dict>` in Python) is far more flexible, especially for styling and per-cell configurations. Use a Script Transform to convert datasets to this format if needed.
*   **Column Configuration:** Remember, if you touch `props.columns` at all, you must define *every* column you want to display.
*   **Performance:**
    *   For large datasets, keep `virtualized` set to `true` (the default). This renders only the visible rows, greatly improving performance.
    *   Enabling `filter.results.enabled` writes the filtered data back to a property, which can hurt performance. Use it only when necessary.
*   **Implicit Parameters:** When using `render: 'view'` or row subviews, remember the implicitly passed parameters (`value`, `rowData`, `row`, etc.). You must create corresponding View Params on the target view to receive and use them. For subviews, the parameter must be an object named `value`.
*   **Styling Precedence:** Style properties can be set at many levels. The most specific style wins. For example, `columns.style` will be overridden by `columns.header.style` for the header cell. Cell-level styles defined in `props.data` are the most specific of all.
*   **Committing Edits:** Remind users they must press `Enter` or `Return` to commit a cell edit. Clicking away or pressing `Esc` will cancel it.
