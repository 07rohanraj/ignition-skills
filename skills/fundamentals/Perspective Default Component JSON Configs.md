# Perspective Default Component JSON Configs

## Description

This document describes the JSON configuration schema for Ignition Perspective Views, detailing the structure of a View and its nested components. It explains the purpose of key properties such as `type`, `meta`, `position`, and `props`, providing the necessary information to interpret, navigate, and programmatically modify the configuration of components within a View.

## Documentation

# Instructions
This document provides instructions for an LLM on how to interpret and manipulate the JSON configuration for Perspective View components in Ignition.

### Introduction to Perspective JSON

Every View in Perspective is defined by a JSON object. This object describes the View's own properties and contains a nested structure of all the components within it. The top-level object for a View has four main keys: `custom`, `params`, `props`, and `root`.

*   `custom`: An object for user-defined properties on the View itself.
*   `params`: An object containing parameters that can be passed into the View from another resource (like an Embedded View component or a Page URL).
*   `props`: Contains properties specific to the View's root container. For example, the `defaultSize` property sets the default design dimensions of the View.
*   `root`: This is the most important key. It contains the primary container for the View and holds a `children` array, which lists all the components placed on the View.

### The `root` Container and its Children

The `root` object represents the top-level container component of a View. By default, this is a Flex Container (`ia.container.flex`). All other components added to the View are placed inside the `children` array of this `root` object.

Each element in the `children` array is a JSON object that defines a single component.

### Component Object Structure

Every component object within the `children` array has a consistent structure:

*   **`type`**: A string that uniquely identifies the kind of component. For example, `"ia.display.label"` for a Label or `"ia.input.button"` for a Button.
*   **`meta`**: An object for metadata about the component. The most important property here is `name`, which is the unique name of the component within the View. This name is used for scripting and creating bindings.
*   **`position`**: An object that defines the component's size and location within its parent container. The properties available under `position` are determined by the `type` of the parent container. For a Flex Container, you will see properties like `basis`, `grow`, and `shrink`. For a Coordinate Container, you will see `x`, `y`, `height`, and `width`.
*   **`props`**: An object containing all the configurable properties specific to that component type. This is where you'll find settings like a Label's `text`, a Table's `data`, or a Chart's `series`.
*   **`custom`**: An object where user-defined custom properties for that specific component can be stored.

### Property Categories Explained

Component properties are organized into categories, which you can see in the Designer's Property Editor. In the JSON, these correspond to the different objects within a component's definition.

*   **`props`**: This object holds the main configuration properties that determine the component's appearance and functionality. The specific properties found here are unique to each component type. For example, an Icon component has a `path` property to define which icon to show, while a Table has a `data` property to hold its rows and columns.
*   **`position`**: These properties control where the component is located inside its parent container. They are not intrinsic to the component itself but are assigned by the container.
*   **`meta`**: These are properties defined by the Perspective module for all components. They include:
    *   `name`: The component's name.
    *   `visible`: A boolean that determines if the component is shown.
    *   `tooltip`: An object to configure a hover tooltip.
    *   `contextMenu`: An object to configure a right-click context menu.
*   **`custom`**: This is the designated safe location for you to add your own custom properties to a component. These properties act like variables that can store information and be accessed by scripts or bindings.
*   **`params`**: This category is **only available on the View itself**, not on individual components within the View. It is found at the top level of the JSON. These properties are used to pass information *into* the View.

### Data Types in JSON

All property values are encoded as JSON. For convenience, the Designer provides special interfaces for certain types, but in the raw JSON, they are one of the following:

*   **Value**: A primitive type.
    *   **String**: A sequence of characters, enclosed in double quotes (e.g., `"Hello, World!"`).
    *   **Number**: An integer or floating-point number (e.g., `101` or `3.14`).
    *   **Boolean**: A `true` or `false` value.
*   **Object**: A collection of key-value pairs, enclosed in curly braces `{}`. The `props` or `style` properties are good examples of objects.
*   **Array**: An ordered list of values, enclosed in square brackets `[]`. A Table's `data` property is an array of objects, where each object represents a row.

### Advanced Property Concepts

*   **Property Persistence**:
    *   **Persistent**: By default, most properties are persistent, meaning their configured value is saved with the View and loaded when a session starts.
    *   **Transient**: A property can be marked as non-persistent (transient). Its value is not saved with the view. This is the default behavior for properties that have a binding, since the binding will provide the value at runtime. User-created custom properties that are not marked as persistent will be lost when the Designer is closed.
*   **Property Access Control**:
    *   Properties can have access levels (`Public`, `Protected`, `Private`) to restrict interaction from browser-side developer tools. This is a security feature to prevent malicious scripts from reading or writing to sensitive component properties.
    *   `Public`: (Default) The property can be read from and written to by scripts in the browser.
    *   `Protected`: The property can be read, but write attempts from the browser are ignored. Only backend sources like bindings or server-side scripts can change the value.
    *   `Private`: The property cannot be read or written to from the browser.

### **Helpful Tips for the LLM**

*   **Finding a Component**: To locate a specific component to modify, traverse the `root.children` array and look for an object where `meta.name` matches the target component's name.
*   **Adding a New Component**: To add a component, you must append a new JSON object to the `root.children` array. This new object needs the required keys: `type`, `meta` (with a `name`), `props`, and `position`. Use the provided schema as a template for the default properties of the component type you are adding.
*   **Identifying Component Type**: The `type` property is the definitive identifier for what a component is (e.g., `ia.input.text-field`, `ia.display.table`).
*   **Custom Properties are Safest**: When asked to add a new property to a component for storing data, always add it to the component's `custom` object. This prevents accidental conflicts with built-in, sometimes hidden, properties in `props` or `meta`.
*   **Position Depends on Parent**: Remember that the fields inside the `position` object (`basis`, `grow`, `shrink`, `x`, `y`, etc.) depend entirely on the `type` of the parent container component (the component that holds the target component in its `children` array).
*   **Default Values are Your Template**: The provided JSON schema shows the default configuration for a brand new component of each type. Use this as a starting point when creating new components.
*   **Modifying Arrays**: To add an item to a property that is an array (e.g., adding a new series to a chart, a new column to a table, or a new item to a menu), find the array property (like `props.series` or `props.items`) and append a new object that conforms to the structure of the existing elements in that array.
*   **Bindings and Scripts**: The static JSON configuration does not represent active bindings or scripts with their logic. A property with a binding will appear as a JSON object describing the binding's configuration, not the resulting value of the binding. Modifying bindings or scripts is a more complex task that may be outside the scope of simple JSON manipulation.
*   **Containers have Children**: Remember that container components (like Flex, Column, Tab, etc.) are the only components that will have a `children` array. Their purpose is to hold other components. The structure is recursive. For example, you can have a Flex Container inside the `root` Flex Container, and that inner container will have its own `children` array.
