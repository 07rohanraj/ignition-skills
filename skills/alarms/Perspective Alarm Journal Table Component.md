# Perspective Alarm Journal Table Component

## Description

These instructions cover the configuration and usage of the Perspective Alarm Journal Table component for displaying historical alarm data. This guide details how to connect the component to a journal, apply filters based on time and alarm properties, and customize the table's appearance, including its columns, row styles, and responsive layouts.

## Documentation

# Instructions
This document provides instructions and tips for an LLM on how to use and configure the **Perspective Alarm Journal Table** component in Ignition.

### Instructions

The Perspective Alarm Journal Table is used to display historical alarm data from a configured alarm journal. It allows for detailed filtering, sorting, and customization of its appearance and behavior.

#### 1. Basic Configuration

*   **Connecting to a Journal:** The most critical first step is to connect the component to an alarm journal. Set the `name` property to the name of the Alarm Journal you have configured in the Ignition Gateway (e.g., "Journal").
*   **Setting the Time Range (`dateRange`):** You have two modes for displaying alarms over time, controlled by `dateRange.mode`:
    *   **Realtime Mode (`realtime`):** This is the default mode. It continuously polls for recent alarms.
        *   To configure the time window, set `dateRange.realtime.interval` (an integer) and `dateRange.realtime.unit` (e.g., "hours", "days", "minutes").
        *   By default, it shows the last 8 hours.
    *   **Historical Mode (`historical`):** This mode shows alarms from a fixed time period.
        *   Set `dateRange.historical.startDate` and `dateRange.historical.endDate` to define the window. These values can be millisecond timestamps or date strings.

#### 2. Filtering Alarms

The `filter` property object offers several ways to refine which alarms are displayed.

*   **Filter by State (`filter.events`):** Set the boolean flags under this property to `true` or `false` to show or hide alarms based on their state. The available states are `active`, `acked`, and `cleared`. `system`, `enabled`, and `disabled` events can also be filtered.
*   **Filter by Priority (`filter.priorities`):** Set the boolean flags for `critical`, `high`, `medium`, `low`, and `diagnostic` to `true` or `false` to control which alarm priorities are visible.
*   **Filter by Text (`filter.text`):** Provide a string to this property to filter alarms based on any of their visible string-based properties.
*   **Filter by Path (`filter.conditions`):** For more specific, gateway-level filtering:
    *   `source`: Filter by the alarm's source path.
    *   `displayPath`: Filter by the alarm's display path.
    *   Both `source` and `displayPath` support the wildcard character (`*`) and can accept multiple paths separated by commas.
    *   `provider`: Filter for alarms from a specific alarm provider.

#### 3. Customizing Columns

You can control the appearance, order, and visibility of all columns.

*   **Standard Columns (`columns`):** This property contains an object for each standard alarm field (e.g., `eventTime`, `source`, `priority`). For each column, you can configure:
    *   `enabled`: A boolean to show or hide the column.
    *   `width`: The proportional width of the column.
    *   `strictWidth`: If `true`, the `width` becomes a fixed pixel value.
    *   `order`: A number that determines the column's position relative to others.
    *   `sort`: The default sort order for the column (`ascending`, `descending`, or `none`).
*   **Associated Data Columns (`columnsAssociated`):** To display custom data that has been stored with an alarm (Associated Data), you must configure it here.
    *   This property is an array of objects. Add a new object to the array for each associated data column you want to display.
    *   In each object, set the `field` property to the exact name of the associated data property you want to show. **This field name must not match any of the standard column names.**
    *   You can also configure `enabled`, `width`, `strictWidth`, `sort`, and `order` for these columns.

#### 4. Styling and Appearance

*   **Row Styles (`rowStyles`):** You can change the appearance of rows based on the alarm's state and priority.
    *   The `rowStyles` property has objects for `active`, `acked`, and `cleared` states.
    *   Within each state, you can define a `base` style and/or specific styles for each `priority`.
    *   You can set properties like `backgroundColor`, `color`, `fontWeight`, or assign a pre-defined Style Class.
*   **Toolbar (`toolbar`):** You can show or hide the entire toolbar or individual buttons within it using the boolean properties under the `toolbar` object (e.g., `enabled`, `enableFilter`, `enableDateRange`).
*   **Header and Pager:**
    *   `enableHeader`: Set to `false` to hide the column header row.
    *   `pager.enabled`: Set to `false` to disable the pagination controls at the bottom of the table.
    *   `pager.hide`: Set to `true` to make the pager controls invisible while keeping pagination active.
*   **Date Formatting:** Use the `dateFormat` property to specify a format for how dates and times are displayed (e.g., "MM/DD/YYYY HH:mm:ss").

#### 5. Advanced Filtering with Scripts

*   For complex filtering logic that can't be achieved with the standard filter properties, you can use the `filterAlarm` extension function.
*   This function is called for every alarm event before it is displayed.
*   It receives one argument, `alarmEvent`, which is an object representing the alarm. You can inspect its properties using `alarmEvent.get('propertyName')`.
*   The script must return `True` to include the alarm in the table or `False` to exclude it.
*   Example Implementation:
    ```python
    # This will only show alarms where the name includes the word "Tank"
    if "Tank" in alarmEvent.get('name'):
        return True
    else:
        return False
    ```

### Helpful Tips

*   **Prerequisite:** Before this component can function, an **Alarm Journal** must be configured in the Ignition Gateway. The component's `name` property must match the name of that journal.
*   **Default State:** When you first drag an Alarm Journal Table into a view, it will be configured by default to show alarm events from the last 8 hours in `realtime` mode.
*   **Performance Considerations:**
    *   The table uses a shared polling engine in `realtime` mode to reduce the performance impact on the Gateway when multiple clients are viewing the same data.
    *   If you need to access the filtered alarm data from a script or binding, you must set `filter.results.enabled` to `true`. Be aware that the schema notes this may cause a performance decline, so only enable it when necessary. The data will then be available in the `filter.results.data` array property.
*   **Wildcard Usage:** The `filter.conditions.source` and `filter.conditions.displayPath` properties support the `*` wildcard for partial path matching, making it easier to filter groups of alarms.
*   **User Interaction:** End-users can interact with the table directly in a session:
    *   **Reorder Columns:** If `dragOrderable` is true, users can drag column headers to change their order.
    *   **Resize Columns:** Users can drag the edge of a column header to resize it.
    *   **Sort Columns:** Clicking a column header will sort the data by that column. Clicking again reverses the sort order.
    *   **Show/Hide Columns:** Right-clicking the table header allows users to show or hide columns.
*   **Associated vs. Standard Columns:** Remember to use the `columns` property for configuring built-in alarm fields and the `columnsAssociated` array for displaying custom data stored with the alarm. The `field` names in `columnsAssociated` cannot overlap with standard field names.
*   **Responsive Layout:** For views that need to work on mobile devices, set `responsive.enabled` to `true`. This will convert the table rows into a card-based layout when the component's width is less than the `responsive.breakpoint` value. Some features, like column headers, are not applicable in this layout.
