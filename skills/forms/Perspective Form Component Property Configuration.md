# Tips
Return only the json configuration of the form's props (property tree), and nothing else. Do not wrap the props tree's json with any extra text. Perspective's form json follows a specific schema, do not stray from that schema.

This is for the Ignition industrial software platform, specifically with an application using the Perspective visualization module.

Perspective Form components use a property tree that is basically JSON.

Form components have a 'props' tree that includes all of the configuration properties for that component.

Documentation for Perspective's Form component is here: https://www.docs.inductiveautomation.com/docs/8.3/appendix/components/perspective-components/perspective-input-palette/perspective-form

Form components should have columns.align set to stretch so that they take up the whole width of the column, otherwise it won't look as good.

Form component widgets need the corresponding schema as part of their property structure. For example, a 'text' widget has a 'text' object as a property (using key 'text') which describes how it should appear and behave. Without that special property, the widget will not appear.

Checkbox and Toggle widgets require the 'type' to be set to either boolean or string within their respective special properties (alongside 'options').

To correctly configure a field within a Perspective Form component, you must modify the props property with the specific JSON structure the component expects.

    Core Structure: The configuration is a nested hierarchy. The path to an individual field's definition is: columns -> items (Array) -> rows -> items (Array) -> widgets (Array)
    Widget Definition: Each object inside the widgets array defines a single form field (e.g., an input).
        "type": Defines the kind of input (e.g., "email", "text").
        "id": A unique name for the field.
        "label": An object containing the visible label, like {"text": "Your Label"}.
    Validation and Field-Specific Properties:
        Configuration specific to a widget's type (like validation rules for an email input) must be nested inside an object whose key matches the type.
        For example, if type is "email", you must create an email object at the same level.
        The path to enable the "required" constraint for an email field is: email -> validation -> constraints -> required: { "enabled": true }.
> Mistake to Avoid: Do not provide a flattened or simplified JSON structure. The full columns/rows/widgets hierarchy is mandatory. Remember that validation rules are not at the top level of the widget but are nested within a block specific to the widget's type (e.g., the email block).

When creating a Perspective Form component configuration, the root is a JSON object. The primary key for defining the visual layout of the input fields is "columns". Another key, "actions", is used for configuring the submit and cancel buttons.

Key Principles:

    Valid Top-Level Keys: The root object can include "columns", "actions", "data", "style", and other keys as defined by the component's model. Do not invent keys like "header" or wrap the entire structure in an unnecessary "form" object.
    Field Layout: The visual structure of the form fields must follow the columns -> items -> rows -> items -> widgets hierarchy.
    Buttons/Actions: All button configuration, including text, alignment, and styling for "submit" and "cancel" actions, must be defined within the top-level "actions" object.
